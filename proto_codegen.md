# Protocol Codegen Audit

## Verdict

The direct `tag struct` protocol path is already close to the ideal lowering:

- one table per protocol member
- one table per protocol getter
- one table per protocol setter
- a tiny helper that loads args, reads `__tag`, and does `call_indirect`
- a tiny per-implementer thunk that only `ref.cast`s and then does `call`, `struct.get`, or `struct.set`

The main place where the lowering is not yet "bare table" is `tag type` parent dispatch. Parent-typed protocol calls currently go through a generated `ref.test` / `ref.cast` variant ladder before they ever reach the table. That is extra control flow and it is the biggest miss against the stated goal.

There are also two smaller transparency issues:

- concrete getter sugar can bypass the protocol table entirely
- getter/setter-only protocols can auto-enroll matching tagged types without explicit conformance

Protocols should be treated as syntax for Wasm tables and `call_indirect`, not
as a place for the compiler to invent extra semantics. Utu is strongest when it
brings Wasm constructs to a usable syntax instead of layering on a second object
model.

## What Is Good Today

### 1. One table per member really is how direct tagged-struct dispatch works

`watgen.js:536-575` builds one slice per readable protocol member and `watgen.js:588-627` does the same for setters. Each slice has its own:

- function type
- trap thunk
- table
- elem segment

For a direct tagged struct, the helper emitted by `watgen.js:1117-1127` is exactly the shape we want:

```wasm
(func $__utu_proto_dispatch_measure_measure_... (param $self (ref $Box)) (result i32)
  local.get $self
  local.get $self
  struct.get $Box $__tag
  call_indirect $__utu_proto_table_measure_measure (type $__utu_proto_sig_measure_measure)
)
```

The same is true for setters:

```wasm
(func $__utu_proto_set_dispatch_counter_ops_value_... (param $self (ref $Counter)) (param $arg1 i32)
  local.get $self
  local.get $arg1
  local.get $self
  struct.get $Counter $__tag
  call_indirect $__utu_proto_set_table_counter_ops_value (type $__utu_proto_set_sig_counter_ops_value)
)
```

That is clear and transparent. There are no hidden guards in the direct helper path.

### 2. The thunk layer is minimal and understandable

`watgen.js:1099-1114` emits thunks that only do:

- `ref.cast` to the concrete receiver type
- `call` for a real method implementation
- `struct.get` for a synthetic getter
- `struct.set` for a synthetic setter

Example synthetic getter thunk:

```wasm
(func $__utu_proto_thunk_counter_ops_value_... (type $__utu_proto_sig_counter_ops_value) (param $self eqref) (result i32)
  local.get $self
  ref.cast (ref $Counter)
  struct.get $Counter $value
)
```

Example synthetic setter thunk:

```wasm
(func $__utu_proto_set_thunk_counter_ops_value_... (type $__utu_proto_set_sig_counter_ops_value) (param $self eqref) (param $arg1 i32)
  local.get $self
  ref.cast (ref $Counter)
  local.get $arg1
  struct.set $Counter $value
)
```

This is about as direct as a shared table design can be. The `ref.cast` is not weird hidden behavior; it is the necessary bridge from the shared `eqref` table signature to the concrete implementation type.

### 3. Missing entries are explicit trap slots, not silent fallbacks

`watgen.js:547`, `watgen.js:599`, and `watgen.js:1095-1097` fill unimplemented tag slots with named trap thunks that do only `unreachable`.

That is good: a bad dispatch does not silently degrade into another path.

## Findings

### Finding 1: `tag type` parent dispatch is not a bare table lowering, and the ref-test ladder must go

Severity: high

The parent helper path generated by `watgen.js:630-680` does not read a tag and jump through a table. Instead it records a `variantDispatch` list, and `watgen.js:1119-1155` lowers that into a recursive `ref.test` / `ref.cast` ladder.

That ladder absolutely must go. If the source asked for a protocol call, then
the lowering should be a protocol call: load the tag and do `call_indirect`
through the member's table. If the user did not ask for RTT checks, the
compiler must not invent RTT checks on their behalf.

Current emitted shape for a parent-typed method call:

```wasm
(func $__utu_proto_dispatch_measure_measure_... (param $self (ref $Shape)) (result i32)
  local.get $self
  ref.test (ref $Box)
  (if (result i32)
    (then
      local.get $self
      ref.cast (ref $Box)
      call $__utu_proto_dispatch_measure_measure_...
    )
    (else
      local.get $self
      ref.test (ref $Square)
      (if (result i32)
        (then
          local.get $self
          ref.cast (ref $Square)
          call $__utu_proto_dispatch_measure_measure_...
        )
        (else
          unreachable
        )
      )
    )
  )
)
```

The same extra ladder exists for parent-typed getters and setters.

Why this exists:

- `watgen.js:1062-1069` emits tagged sum roots as `(type $Parent (sub (struct)))`
- the hidden `__tag` field is only added to each variant, not to the parent
- because a `(ref $Parent)` cannot `struct.get $Parent $__tag`, the compiler falls back to RTT checks

This directly conflicts with the spec language in `documentation/spec.typ:252-253`, which says lowering uses tag-indexed `call_indirect` helpers.

Recommended fix:

- put `__tag` on the tagged sum root type itself
- make variants subtype that root and add only their own payload fields
- then emit the same helper shape for parent types that already exists for direct tagged structs

Target shape:

```wasm
(type $Shape (sub (struct
  (field $__tag i32)
)))
(type $Box (sub $Shape (struct
  (field $width i32)
  (field $height i32)
)))

(func $__utu_proto_dispatch_measure_measure_shape (param $self (ref $Shape)) (result i32)
  local.get $self
  local.get $self
  struct.get $Shape $__tag
  call_indirect $__utu_proto_table_measure_measure (type $__utu_proto_sig_measure_measure)
)
```

That would remove the variant ladder entirely and make parent dispatch match the direct tagged-struct path. This should be treated as required cleanup, not an optional optimization.

### Finding 2: concrete getter sugar can bypass the protocol table today, and that is the wrong lowering

Severity: medium

`watgen.js:1474-1485` only calls a getter helper when there is no direct field:

```js
if (getterHelper && !directField) {
    out.push(`call $${getterHelper}`);
    return;
}
out.push(`struct.get $${objectType} $${field.text}`);
```

That means a concrete tagged struct like `Counter` with a protocol getter for `value` will compile:

- `CounterOps.value(counter)` through the protocol table
- `counter.value` as a plain `struct.get`

Setters do not behave this way. `watgen.js:353-365` prefers the setter helper whenever one exists, even when the field is direct.

So getter/setter lowering is currently asymmetric:

- setter sugar prefers protocol dispatch
- getter sugar prefers direct field access

This is not just a style issue. It means the protocol getter syntax does not
consistently mean "read through the protocol table". The source form stops
mapping cleanly onto the Wasm construct it is supposed to represent.

How the bypass happens:

- `objectType` is inferred from the receiver
- `directField` is found on that concrete type
- `getterHelper` is also found for the same member name
- because `directField` exists, the code takes the `struct.get` branch and never calls the helper
- the helper is the only path that performs the table lookup

That behavior should be removed. A protocol getter should always lower through
its dedicated member table, because that is what a protocol is. If a member is
declared in a protocol, every use of that protocol member should mean
table-backed `call_indirect`, not "maybe a table, maybe a direct field read".

### Finding 3: getter/setter-only protocols auto-enroll matching tagged types

Severity: medium

`watgen.js:775-779` walks every tagged type in the program and adds it as an implementer when all members are field-backed and the fields happen to match. `expand.js:1215-1225` also has an explicit getter-backed fallback for protocol calls.

That means tables and helpers can be synthesized for types that never explicitly wrote `: Proto` and never wrote `fun Proto.member(...)`.

This is convenient, but it is hidden behavior from a lowering-audit perspective. It affects:

- which types get dispatch slots
- which helpers exist
- which explicit protocol calls are accepted

Recommendation:

- remove this implicit conformance path, or at minimum treat it as a bug until the language explicitly blesses it
- protocol table population should come from explicit protocol surface, not shape-matching guesses by the compiler

This falls under the same principle as the ref-test ladder: we are not in the
business of inventing constructs. We are in the business of bringing Wasm
constructs to a usable syntax.

### Finding 4: tables are global-tag-sized, not implementer-sized

Severity: low

`watgen.js:521-547` and `watgen.js:578-600` size each protocol table as `maxTag + 1`, where `maxTag` is the highest tag assigned anywhere in the module.

That means unrelated tagged types create holes in every protocol table. Example from the compiled `examples/ci/codegen_protocols.utu` WAT:

- `Measure.measure` has a table of size 4
- only `Box` and `Square` actually implement it
- `Rect` and `Counter` occupy trap slots

This is not hidden behavior. It is explicit and easy to inspect in the emitted `elem` segment. It is also a reasonable tradeoff if the priority is a direct tag-to-table-index mapping.

I would not change this unless table size becomes a real problem. It keeps dispatch simple.

## Test Coverage Notes

The current tests are good, but they do not fully pin the "bare table" property:

- `scripts/test-modules.mjs:1007-1041` asserts the direct tagged-struct path emits a hidden tag, a dedicated table, an elem segment, and `call_indirect`
- `scripts/test-modules.mjs:478-521` checks tagged-sum parent dispatch behaviorally, but not structurally
- there is no codegen assertion that parent-tagged-sum dispatch avoids `ref.test`
- there is no codegen assertion for concrete getter sugar choosing helper-vs-direct-field

If this area is going to be tightened, I would add WAT-shape tests for:

- parent `tag type` method dispatch contains no `ref.test`
- parent `tag type` getter dispatch contains no `ref.test`
- parent `tag type` setter dispatch contains no `ref.test`
- whichever getter policy is chosen for concrete tagged structs

## Bottom Line

The direct tagged-struct lowering is already clean and close to ideal.

The biggest remaining issue is structural: tagged-sum parent types do not carry the hidden tag field, so parent dispatch cannot lower straight to `struct.get $__tag + call_indirect`. Fixing that layout would remove the main extra checks and make methods, getters, and setters all look like true bare-table lowering from the parent type as well.

Protocols should be documented and implemented as exactly this: named syntax for
tables and `call_indirect`. A protocol member is not an abstract "virtual"
thing that sometimes lowers to a table and sometimes does something else. The
table is the construct.
